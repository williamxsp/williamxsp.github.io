Teste de Mudanas

Os testes sobre mudanas so tratados de uma forma especial, pois: 1) No precisamos testar todo o sistema novamente quando mudamos um componente precisamos testar apenas os componentes relacionados, observando as matrizes de integridade referencial. 2) Podemos reaproveitar testes j elaborados anteriormente para reavaliar programas que no foram modificados mas sero re-testados, focando esforos em construir novos casos de testes somente para aquilo que novo.

Testes Estruturais

WMC Weight of Methods in a Class

Mtodos ponderados por Classe (WMC)
CMi = nmero de mtodos da Classe i
WMC = nmero de mtodos resultante da soma dos mtodos de todas as Classes

 

Anlise

Quanto maior o nmero de mtodos, mais complexa a Classe em termos do seu desenvolvimento e manuteno deve ser verificada a possibilidade de especializao para Classes com uma quantidade de mtodos muito acima do padro de outras Classes do sistema, ou que representa um grande percentual do total de mtodos do projeto de sistema. Poucos mtodos podem indicar uma especializao desnecessria de Classe.

 

DIT - Depth Inside the Tree

Profundidade da rvore de herana

Avalia o nmero de nveis hierrquicos abaixo de uma Classe. O indicador calculado para cada Classe modelada.

 

Anlise

Se o sistema tiver quase todas as Classes com profundidade 0, a Herana (reuso) est comprometida. Profundidade muito grande pode apontar excesso de especializao de Classes o nmero de Classes pode ser reduzido e os objetos tipificados.

 

NOC - Number of Chidren

Nmero de Filhos

Avalia o nmero de Classes que so herdeiras diretas (filhos diretos) de uma Classe. Calculado Classe a Classe

Anlise

Classes com apenas 1 filho indicam que temos uma especializao de Classes excessiva, gerando complexidade desnecessria para o sistema.

 

LCOM - Level of Cohesion of Method

Coeso em mtodos

Avalia o quanto um mtodo de um Classe depende de outros Mtodos de outras Classes para resolver a sua lgica.

 

Uma Classe coesa se os seus Mtodos so coesos. Dois mtodos de uma Classe X so coesos se utilizam os atributos da Classe X e no de outra.

 

LCOM = (nmero de Atributos usados no Mtodo) / (nmero de atributos utilizados da Classe do Mtodo pelo Mtodo)

 

Calculado Mtodo a Mtodo

Anlise

Se algum mtodo da Classe utilizar atributos que no so dela, ele avaliado como no coeso. O nvel de coeso dado pelo nmero de atributos utilizados pelo mtodo sobre o total de atributos da Classe. Se a diviso der 1, a coeso total. Quanto maior o valor obtido da diviso, pior a coeso. Se um mtodo no est totalmente coeso, ele rompe com o princpio de Encapsulamento dos Objetos.

 

CBO - Coupling Between Objects

Acoplamento entre classes

Avalia o quanto uma Classe acessa Mtodos e Atributos de outra Classe. Uma Classe est acoplada a outra quando seus mtodos utilizam mtodos da outra Classe e/ou operam diretamente sobre atributos da outra Classe. CBO = nmero Mtodos de outras Classes + Nmero de Atributos de outras Classes que so usados por uma Classe avaliada. Calculado Classe a Classe

Anlise

 

se o nmero de acoplamento for alto entre duas Classes, elas podem eventualmente ser Generalizadas ou seja, Unificadas. OU Se uma Classe aciona muitas outras Classes, isso pode indicar a necessidade de uma reviso do seu tipo e contedo (de Entidade para Controle, por exemplo) e portanto, pode implicar em reengenharia do projeto.

 

RFC - Response For a Class

Resposta para uma Classe

Avalia quantas vezes um Mtodo de uma Classe chama Mtodos de outra Classe. Indica o nvel de dependncia que uma Classe tem das outras para resolver um processo de negcio (rotina da aplicao por completo). RFC = nmero de vezes em que ocorrem acionamentos (mensagens) que os Mtodos da Classe fazem para Mtodos de outras Classes. Calculado Classe a Classe

Anlise

 

O nvel de RFC baseado na quantidade de mtodos que uma Classe aciona de outras. Se uma Classe aciona demasiadamente uma nica outra Classe, isso aponta que seus Mtodos no lhe garantem auto suficincia, apontando uma provvel necessidade de Agregao de Classes (Generalizao) ou incorporao da lgica de um mtodo de outra classe no mtodo chamador, (reviso na lgica dos mtodos, mudando escopos).

Nos testes unitrios, devemos mascarar as linhas de comando de uma procedure, funo ou mtodo de Classe de objetos, simulando o retorno da funo chamada com retornos corretos, de forma a permitir a avaliao da aplicao desenvolvida sem a contaminao do resultados das rotinas externas acionadas. Quando precisamos fazer TESTES DE INTEGRAO, devemos ir retirando os comentrios dos acionamentos das rotinas externas, um aps o outro, de forma a validar isoladamente os retornos de cada rotina, facilitando a deteco de falhas.

LowerCase

Partem de descries em um texto estruturado em lingua natural e geram cdigos e bancos de dados em linguagens de programao e sistemas gerenciadores de bancos de dados.

 

MiddleCase

Partem de modelos lgicos tcnicos (projeto lgico), em especial modelos de bancos de dados e prottipos de telas e geram os cdigos finais de aplicao.

 

UpperCase

 

Permitem a diagramao lgica do entendimento de negcios (especificao de alto nvel), gerando cdigo de software e bancos de dados partir dos diagramas de alto nvel.

 

I-Case

Possibilitam que a especificao de software ocorra em vrios nveis de modelagem, refinando sucessivamente a soluo.

Incluem: prototipador de interfaces, diagramador de bancos de dados, editor de linguagem de especificao lgica, interpretador ou compilador, depurador, ferramentas de testes e gerador de documentao, ferramentas de avaliao de qualidade, riscos e gesto de projeto. Geralmente possibilitam a engenharia reversa do software.
